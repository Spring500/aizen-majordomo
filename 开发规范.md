# 开发规范

本项目人与 AI agent 共同协作，遵循统一的**提交信息规范**与**开发工作流**。两者均由 husky hook 在本地硬性校验。本文件是单一事实源，`README.md` 与 `AGENTS.md` 均引用此处。

---

# 第一部分 · 提交信息规范（commit message）

所有 commit message 必须遵守以下规范，由 `.husky/commit-msg`（调用 `scripts/verify-commit-msg.mjs`）在提交时**硬性校验**，不合规直接拒绝。

## 一、格式模板

```
<类型>: <标题>

意图：<为什么做这次提交——要解决的问题或达成的目标>

主要修改：
- <修改点一>
- <修改点二>
```

示例：

```
功能: 增加令牌鉴权中间件

意图：让 agent 通过 Bearer 令牌访问 API，并按角色 scope 限权。

主要修改：
- 新增 src/middleware/auth.ts，解析并校验令牌
- tokens/roles 表查询与 scope 比对
- 在 /cards 路由挂载中间件
```

## 二、类型（全中文，九选一）

| 类型 | 含义 |
|---|---|
| `功能` | 新增功能 / 端点 / 能力 |
| `修复` | 修复 bug 或错误行为 |
| `重构` | 不改变外部行为的代码结构调整 |
| `文档` | 仅改动文档（README、规范、注释文档等） |
| `测试` | 新增或调整测试 |
| `构建` | 构建、依赖、工具链、脚本、CI 相关 |
| `性能` | 性能优化（不改变功能语义） |
| `样式` | 代码格式 / 风格调整（不影响逻辑） |
| `杂项` | 其他无法归入上述类别的零碎改动 |

需要时可加范围：`功能(auth): 标题`，范围放在类型后的圆括号里。

## 三、规则清单（逐条）

1. **首行格式为 `类型: 标题`**：类型必须是上表九种之一；类型与标题之间用一个半角冒号加一个空格 `: ` 分隔；可选范围写作 `类型(范围):`。
2. **标题用中文描述本次改动**，简明扼要，控制在 50 个字符以内；标题末尾不加句号。
3. **首行之后必须空一行**，再写正文。
4. **正文必须包含「意图」段**：以 `意图：` 开头（全角或半角冒号均可），说明**为什么**做这次提交，而不是复述改了什么。
5. **正文必须包含「主要修改」段**：以 `主要修改：` 开头，其下至少一条以 `- ` 开头的要点，逐条列出**主要改了什么**。
6. **意图与主要修改必须分开写**：意图回答「为什么」，主要修改回答「改了什么」。
7. **禁止任何模型署名**：不得出现 `Co-Authored-By`、`Generated with ...`、机器人 emoji 等标识生成模型的内容。原因：一件工作可能跨多个模型完成，标注模型反而干扰统计与归因。
8. **全程使用中文叙述**：意图与修改点用中文书写；代码标识符、文件路径、英文术语（如 `Bearer`、`scope`、`SQLite`）可按原文保留。
9. **合并 / 回退等 git 自动生成的提交**（`Merge ...`、`Revert ...`、`fixup!`、`squash!`）不受本规范约束，hook 自动放行。
10. **类型与文件性质须一致（保障原子性，宽松模式）**：见下节。

## 四、类型与文件性质一致性

为防止「README 混进构建提交」这类破坏原子性的错误，hook 读取本次暂存文件（`git diff --cached`），按性质分类并与首行类型交叉核对。

| 性质 | 命中规则（示例） |
|---|---|
| 文档 | `*.md`、`docs/`、`README`、`CHANGELOG`、`LICENSE` 等 |
| 构建/工具链 | `package.json`、`*lock*`、`pnpm-workspace.yaml`、`tsconfig*.json`、`.husky/`、`scripts/`、`.github/`、`*.config.*`、`.gitignore` 等 |
| 测试 | `*.test.*`、`*.spec.*`、`tests/`、`__tests__/` |
| 代码 | 其余（如 `src/**`） |

校验策略（**宽松**）：

- `文档` / `构建` / `测试` 提交 → 暂存文件必须**全部**是对应性质，否则拒绝。
- `功能 / 修复 / 重构 / 性能 / 样式 / 杂项` → **不校验文件性质**，允许代码、文档、依赖混在一起（如功能提交顺带改 README 或加依赖）。

> 局限：代码类型彼此无法靠文件路径区分，「一个提交=一件逻辑事」的语义原子性仍需靠纪律（小步提交、`git add -p`）与 review 保障。

## 五、反例（会被拒绝）

```
add auth middleware            # ✗ 无中文类型、无意图/主要修改段、纯英文
```
```
功能: 增加鉴权

主要修改：
- 加了中间件                    # ✗ 缺「意图」段
```
```
功能: 增加鉴权

意图：让 agent 能鉴权

主要修改：
- 加了中间件

Co-Authored-By: Claude <noreply@anthropic.com>   # ✗ 含模型署名
```

# 第二部分 · 开发工作流（worktree + 两段式门禁）

核心不变量：**main 只通过「通过全部检查的落地」更新；日常开发在 worktree/分支上进行。**

## 一、在 worktree / 分支上开发，不在 main 上开发

- 每项工作新建分支或 git worktree，在其中编码、提交、自查。
- `main` 上的任何提交都会触发**全量门禁**（见下），因此请勿在 main 上做日常开发；main 只接收落地。

## 二、两段式测试门禁

| 触发点 | 检查范围 | 由谁执行 |
|---|---|---|
| **分支提交**（`pre-commit`，HEAD≠main） | **轻量**：全部快速 Vitest（单元 + HTTP 验收，进程内） | `.husky/pre-commit` |
| **落地 main**（main 上提交，或合并进 main） | **全量**：快速 Vitest + Playwright（网页+集成，待前端）+ 合入消息合规 | `.husky/pre-commit` / `.husky/pre-merge-commit` |

- **分支提交是硬门禁**：测试红则拒绝提交。配合下面的约定，红绿循环不受影响。
- **commit-msg 始终校验**：无论分支还是 main，提交信息都必须合规（第一部分）。

## 三、红绿循环的两个约定

1. **未实现的行为规格用 `it.todo` / `it.skip` 占位**：可先把行为描述清楚并提交（套件仍绿、过门禁）；实现时换成真断言，用 `pnpm test:watch` 看它红→绿，绿了连同实现代码一起提交。这样同时满足「先描述行为」「测试先于代码」「每个提交皆绿」。
2. **私有分支的临时 WIP 可 `--no-verify`**：偶尔存中途快照时用；走 squash 落地时这些 WIP 提交会被压没。除此之外不要用 `--no-verify`。

## 四、落地 main：两种方式都支持

落地在主仓目录（main 检出处）进行，从 worktree 分支合并：

- **squash 落地**：`git merge --squash <分支>` 后 `git commit`。该提交落在 main → `pre-commit` 检测 `HEAD==main` → 跑全量。分支的脏历史/脏 message 被丢弃，只校验最终这一条。适合分支历史杂乱的情况。
- **`--no-ff` 合并落地**：`git merge --no-ff <分支>` → `pre-merge-commit` 跑全量 + 校验**合入范围每条 message** 合规（`scripts/verify-range-messages.mjs`）。保留分支历史，因此分支提交的 message 也必须合规（本就强制）。

> 仓库已设 `merge.ff = false`（由 `prepare` 脚本自动配置），杜绝快进合并跳过 `pre-merge-commit` 闸门。

## 五、本地 hook 的边界（务必知晓）

- 本地 hook 可被 `--no-verify` 绕过——纯本地 git 无法根治。真正不可绕过的强制是**服务端 branch protection + CI**（接入远程后再补）。本地 hook 是「快的第一道防线 + 给 agent 即时反馈」。
- hook 无法区分提交者是人还是 agent，故规则对所有提交者一视同仁。
