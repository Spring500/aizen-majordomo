# 提交信息规范（commit message）

本项目所有 commit message 必须遵守以下规范，并由 husky 的 `commit-msg` hook 在提交时**硬性校验**，不合规将直接拒绝提交。

---

## 一、格式模板

```
<类型>: <标题>

意图：<为什么做这次提交——要解决的问题或达成的目标>

主要修改：
- <修改点一>
- <修改点二>
- ...
```

示例：

```
功能: 增加令牌鉴权中间件

意图：让 agent 通过 Bearer 令牌访问 API，并按角色 scope 限权。

主要修改：
- 新增 src/middleware/auth.ts，解析并校验令牌
- tokens/roles 表查询与 scope 比对
- 在 /cards 路由挂载中间件
```

---

## 二、类型（必须二选一以上其一，全中文）

| 类型 | 含义 |
|---|---|
| `功能` | 新增功能 / 端点 / 能力 |
| `修复` | 修复 bug 或错误行为 |
| `重构` | 不改变外部行为的代码结构调整 |
| `文档` | 仅改动文档（README、规范、注释文档等） |
| `测试` | 新增或调整测试 |
| `构建` | 构建、依赖、工具链、脚本、CI 相关 |
| `性能` | 性能优化（不改变功能语义） |
| `样式` | 代码格式 / 风格调整（不影响逻辑） |
| `杂项` | 其他无法归入上述类别的零碎改动 |

需要时可加范围：`功能(auth): 标题`，范围放在类型后的圆括号里。

---

## 三、规则清单（逐条）

1. **首行格式为 `类型: 标题`**：类型必须是上表九种之一；类型与标题之间用一个半角冒号加一个空格 `: ` 分隔；可选范围写作 `类型(范围):`。
2. **标题用中文描述本次改动**，简明扼要，控制在 50 个字符以内；标题末尾不加句号。
3. **首行之后必须空一行**，再写正文。
4. **正文必须包含「意图」段**：以 `意图：` 开头（全角或半角冒号均可），说明**为什么**做这次提交（解决的问题 / 要达成的目标），而不是复述改了什么。
5. **正文必须包含「主要修改」段**：以 `主要修改：` 开头，其下至少一条以 `- ` 开头的要点，逐条列出**主要改了什么**。
6. **意图与主要修改必须分开写**：意图回答「为什么」，主要修改回答「改了什么」，不要混为一谈。
7. **禁止任何模型署名**：不得出现 `Co-Authored-By`、`Generated with ...`、机器人 emoji 等标识生成模型的内容。原因：一件工作可能跨多个模型完成，标注模型反而干扰统计与归因。
8. **全程使用中文叙述**：意图与修改点用中文书写；代码标识符、文件路径、英文术语（如 `Bearer`、`scope`、`SQLite`）可按原文保留。
9. **合并 / 回退等 git 自动生成的提交**（`Merge ...`、`Revert ...`、`fixup!`、`squash!`）不受本规范约束，hook 自动放行。
10. **类型与文件性质须一致（保障原子性，宽松模式）**：`文档`、`构建`、`测试` 三种「非代码」类型的提交只允许包含对应性质的文件；代码类型（`功能/修复/重构/性能/样式/杂项`）宽松放行，可夹带文档等其他文件。详见下节。

---

## 三-补、类型与文件性质一致性

为防止「README 混进构建提交」这类破坏原子性的错误，hook 会读取本次暂存文件（`git diff --cached`），按文件性质分类，并与首行类型交叉核对。

文件性质分类：

| 性质 | 命中规则（示例） |
|---|---|
| 文档 | `*.md`、`docs/`、`README`、`CHANGELOG`、`LICENSE` 等 |
| 构建/工具链 | `package.json`、`*lock*`、`pnpm-workspace.yaml`、`tsconfig*.json`、`.husky/`、`scripts/`、`.github/`、`*.config.*`、`.gitignore` 等 |
| 测试 | `*.test.*`、`*.spec.*`、`tests/`、`__tests__/` |
| 代码 | 其余（如 `src/**`） |

校验策略（**宽松**）：

- `文档` 提交 → 暂存文件必须**全部**是文档，否则拒绝。
- `构建` 提交 → 必须**全部**是构建/工具链文件，否则拒绝。
- `测试` 提交 → 必须**全部**是测试文件，否则拒绝。
- `功能 / 修复 / 重构 / 性能 / 样式 / 杂项` → **不校验文件性质**，允许代码、文档、依赖等混在一起（如功能提交顺带改 README 或加依赖）。

> 局限：代码类型彼此（功能/修复/重构…）无法靠文件路径区分，「一个提交=一件逻辑事」的语义原子性仍需靠纪律（小步提交、`git add -p`）与 review 保障，hook 只兜住「类型↔文件性质」这一类机械错误。

## 四、反例（会被拒绝）

```
add auth middleware            # ✗ 无中文类型、无意图/主要修改段、纯英文标题
```

```
功能: 增加鉴权

主要修改：
- 加了中间件                    # ✗ 缺「意图」段
```

```
功能: 增加鉴权

意图：让 agent 能鉴权

主要修改：
- 加了中间件

Co-Authored-By: Claude <noreply@anthropic.com>   # ✗ 含模型署名
```

---

## 五、校验由谁执行

提交时 `.husky/commit-msg` 调用 `scripts/verify-commit-msg.mjs` 逐项检查上述规则；不通过即 `exit 1` 阻止提交，并打印具体原因。如需在极特殊情况下临时绕过（不推荐），可用 `git commit --no-verify`。
