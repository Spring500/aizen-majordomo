# aizen-majordomo 路线图

本文档定义产品阶段顺序、阶段范围和验收基线。开发、提交、PR 和 CI 规则见 [../CONTRIBUTING.md](../CONTRIBUTING.md)。

目标：用纵向切片方式尽快做出“基本可用”的版本，再逐步补齐状态、权限、管理、hook 和体验打磨。

## 0. 路线原则

本项目不按“先把所有后端 API 做完，最后再做前端”的方式推进。那种方式工程分层整齐，但太晚才能感受到产品是否真的可用。

本路线图按“可用闭环”推进：每个阶段都交付一段可以实际试用的能力。越早让人和 agent 跑通真实流程，越早能发现交互、状态、权限和信息结构上的问题。

## 1. 文档边界

本文档只定义：

- 阶段顺序。
- 每阶段要交付的能力。
- 每阶段明确不做的内容。
- 自动化测试必须覆盖的行为。
- 人工验收要按什么步骤操作、看到什么结果。

本文档不记录阶段是否已经完成。阶段进度应由 issue、分支、交接文档、实施计划或提交历史记录。除非路线本身改变，不应因为某阶段完成而修改本文档。

每个阶段开始前，后续 agent 应另写实施计划。实施计划负责拆文件、测试用例、提交边界和具体执行步骤。

## 2. 全局门禁和验收的区别

`pnpm test`、`pnpm typecheck`、Playwright 全量验收是门禁，不是单独的业务验收标准。

原因：如果没有写对应测试，命令通过并不能证明行为被覆盖。

因此每个阶段必须同时满足两类要求：

- 自动化测试覆盖要求：告诉实现 agent 要把哪些行为写进单元测试、HTTP 集成测试、端到端测试。
- 人工验收指南：告诉人类如何像真实用户一样操作，并判断产品体验是否成立。

全局门禁仍然必须执行：

- 分支提交前，快速 Vitest 必须通过。
- 落地 main 前，GitHub PR required checks 必须通过：Commit messages + Tests。
- 新增或修改的测试里，每个 `expect` 都必须带中文辅助信息。
- README、交接文档或阶段说明中与实际行为不一致的内容必须同步更新。
- 不得误提交 `data/`、`代理执行笔记/`、`docs/local/` 等本地资料。
- 提交与落地流程必须遵守 [../CONTRIBUTING.md](../CONTRIBUTING.md) 和 [../AGENTS.md](../AGENTS.md)。
- 后续任何涉及前端的阶段，都必须同时考虑横屏/宽屏和竖屏/窄屏布局。核心能力不能在窄屏被删除，只能改为抽屉、面板、弹层或其他适合窄屏的呈现方式；自动化测试或人工验收必须覆盖这种功能对位。

## 阶段 2：基础配置模型

### 目标

在进入人机闭环 MVP 前，先把卡片类型、状态、流转和 hook 的概念关系固定下来，并让现有建卡、编辑能力开始受配置驱动。此阶段不追求完整管理界面。

### 概念关系

- 卡片类型定义一张卡的形态和能力，例如字段、动作、是否需要正式回复、是否带选项。
- 字段能力使用字段定义数组表达；字段定义描述字段本身，例如标识、显示名、类型和必填性。
- 动作定义声明某个操作允许写入哪些字段。内置 `create` 动作用于建卡，内置 `update` 动作用于 `PATCH /cards/{id}`。
- 状态定义一张卡在工作流中的位置，不决定卡片字段。
- 流转定义某类卡在某个来源状态下，可以进入哪个目标状态，以及流转时允许或要求写入哪些字段。
- hook 定义事件发生后的自动动作。自动状态变化应通过 hook 触发已配置流转表达，而不是放进创建、编辑、回复等接口的特殊分支。
- changes 记录事实变化，供 agent 增量读取；hook 和前端视图都应基于这些事实和配置工作。

### 配置存储和初始化决策

- 运行时配置以 SQLite 数据库为事实来源；TypeScript 样例配置只作为默认种子、测试基准和代码审阅入口。
- 数据库存储采用“核心实体表 + JSON 能力字段”的混合模型：卡片类型、状态、流转、hook 或 hook action model 作为核心实体；字段能力、动作能力、匹配条件和动作参数等可存为 JSON。
- 阶段 2 的内置样例配置初始化采用同 id 覆盖更新策略，保证样例配置与代码一致；阶段 7 引入配置管理后，必须改为保护用户管理过的配置。
- 阶段 2 的配置初始化和引用校验采用 fail fast：配置无效时初始化或启动失败，不允许服务带着无效配置继续提供核心能力。

### 样例配置要求

阶段 2 必须产出一套后续开发和测试共用的样例配置。后续阶段提到 `task`、`decision`、`memo` 时，均指这套样例配置中的卡片类型，而不是代码里的固定类型。

| 配置类别 | 样例内容 | 用途 |
|----------|----------|------|
| 卡片类型 | `task`、`decision`、`memo` | 覆盖普通事项、需要人类正式回复的问题、备忘记录三种常用形态。 |
| 通用字段 | `title`、`body`、`priority`、`lane`、`assignee` | 支撑阶段 1 已有看板能力和后续筛选、编辑。 |
| task 字段和动作 | 通用字段、编辑、评论、流转 | 支撑普通事项从创建到完成的演示。 |
| decision 字段和动作 | 通用字段、`options`、`reply`、`replied_by`、`reply` 动作 | 支撑阶段 3 的人机回复闭环。 |
| memo 字段和动作 | `title`、`body`、评论、流转 | 支撑备忘记录和协作上下文演示。 |
| 扩展验收字段 | 至少一个阶段 1 之外且支持基础过滤的字段，例如 `risk_level` | 验证新增配置字段能被读取、写入、编辑和过滤，而不是只覆盖旧固定字段。 |
| 状态 | `default`、`active`、`waiting`、`resolved`、`done` | 支撑阶段 4 的配置化流转验收。 |
| 流转 | `start`、`request_reply`、`submit_reply`、`complete` | 覆盖普通卡片推进、decision 等待回复、提交回复和完成。 |
| hook 动作模型 | `transition`、`webhook`、`script` | 支撑阶段 8 的 hook 执行能力设计；阶段 2 只验证配置表达，不执行 hook。 |

## 每阶段开始前的实施计划要求

每个阶段开始前，负责实现的 agent 应先写实施计划，并至少包含：

- 本阶段目标。
- 范围内功能和范围外功能。
- 需要修改或新增的文件。
- 单元测试覆盖清单。
- HTTP 集成测试覆盖清单。
- 前端组件或端到端测试覆盖清单。
- 人工验收操作指南。
- 提交拆分建议。
- 阶段完成定义。
- 如果本阶段包含前端改动，必须说明横屏/宽屏与竖屏/窄屏布局如何保持功能对位，并列出对应的自动化测试或人工验收点。

实施计划不能把“测试命令通过”当成业务验收本身。测试命令通过只是门禁，真正的验收来自具体行为是否被自动化测试和人工流程覆盖。

实施计划不能弱化本文档的验收标准。如需改动阶段范围，应先向用户说明原因并得到确认。
